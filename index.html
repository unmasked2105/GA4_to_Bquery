Part 01

Setting Up the Connection

GA4 has a native BigQuery export — no ETL tool, no custom connector, no Fivetran needed. Google built this in for free. You configure it once, it runs forever automatically.

1
Confirm you have GA4 (not Universal Analytics)
Native BQ export is GA4-only. If your company is still on Universal Analytics (UA), you'll need a different approach. Check the property — GA4 properties show a number like G-XXXXXXXXXX. UA properties show UA-XXXXX-X.
ℹ GA4 = free BQ export ✗ UA = not free, needs workaround
2
Make sure BigQuery API is enabled in GCP
In your GCP Console, go to APIs & Services → search BigQuery → confirm it's enabled. Also confirm you have a GCP project ready. Since you're already using BQ for other pipelines, this is probably already done.
GCP Console → APIs & Services → BigQuery API → Enable
✓ Likely already done at your internship
3
Link GA4 Property to BigQuery
This is the actual setup step. In GA4 go to Admin, then under the Property column find BigQuery Links. Click Link. Select your GCP project. Choose your BigQuery dataset location — pick the same region as your other BQ datasets (usually US or EU). Mismatched regions cause query errors later.
GA4 Admin → Property Column → BigQuery Links → Link
✓ One time setup, no code needed ⚠ Match region to your existing BQ datasets
4
Choose Export Type: Daily vs Streaming
During linking, GA4 asks which type of export you want.
Daily Export
Table: events_YYYYMMDD
Runs once per day, next morning. ~24hr delay. Free. Good enough for 90% of pipeline work. Use this for your internship pipeline.
Streaming Export
Table: events_intraday_YYYYMMDD
Events land within ~1 min. Has BQ streaming costs. Updated continuously until replaced by daily table next morning. Has occasional duplicates.
✓ For a pipeline — choose Daily. Free + reliable.
5
Verify Data is Landing in BigQuery
After 24–48 hours, open BigQuery console. Look for a new dataset named after your GA4 property ID — format is analytics_XXXXXXXXX. Inside that dataset you should see daily tables appearing. If you see them — you're connected and data is flowing.
BigQuery Console → Your Project → analytics_XXXXXXXXX → events_YYYYMMDD
✓ If tables appear = connection is working ✗ No tables after 48h = check IAM permissions
6
IAM Permissions — If something breaks
GA4 uses a Google-managed service account to write data into your BQ project. If it doesn't have permission, data won't land. The service account is firebase-measurement@system.gserviceaccount.com. It needs BigQuery Data Editor role. Usually auto-configured during linking — only fix this if data isn't appearing after 48 hours.
GCP IAM → Grant: firebase-measurement@system → BigQuery Data Editor
⚠ Only needed if data is not appearing
Part 02

How Data Lands — Schema

This is the most important thing to understand before you write a single line of pipeline code. GA4's schema has one design decision that trips up every beginner.

The UNNEST Pattern
GA4 stores event parameters as an ARRAY of STRUCTs called event_params. Instead of separate columns for each parameter, every param is a key-value pair inside a nested array. To access any parameter (like page_url, session_id, engagement_time), you must UNNEST the array and filter by key name. This is the single most important SQL pattern in GA4 BQ work.
Raw in BigQuery
event_params: [
  {key: "page_location",
   value: {string_value:
    "example.com/home"}},
  {key: "ga_session_id",
   value: {int_value: 1234567}},
  {key: "page_title",
   value: {string_value: "Home"}}
]
→
After UNNEST in SQL
page_location: "example.com/home"
ga_session_id: 1234567
page_title: "Home"

▸ UNNEST template — copy and modify for your pipeline

UNNEST Pattern — Flatten event_params into columns SQL · BigQuery 
-- Template: flatten GA4 nested params into a clean table
-- Replace project/dataset/date with your actual values
SELECT
    -- Direct columns (no UNNEST needed)
    event_date,
    event_name,
    user_pseudo_id,
    TIMESTAMP_MICROS(event_timestamp)     AS event_time,
    geo.country,
    geo.city,
    device.category                         AS device_type,
    traffic_source.source,
    traffic_source.medium,

    -- UNNEST pattern for string params
    (SELECT value.string_value
     FROM UNNEST(event_params)
     WHERE key = 'page_location')          AS page_url,

    (SELECT value.string_value
     FROM UNNEST(event_params)
     WHERE key = 'page_title')             AS page_title,

    -- UNNEST pattern for integer params
    (SELECT value.int_value
     FROM UNNEST(event_params)
     WHERE key = 'ga_session_id')          AS session_id,

    (SELECT value.int_value
     FROM UNNEST(event_params)
     WHERE key = 'engagement_time_msec')    AS engagement_ms

FROM `your_project.analytics_XXXXXXXXX.events_*`
WHERE
    -- ALWAYS filter _TABLE_SUFFIX — avoids scanning all tables
    _TABLE_SUFFIX = '20240115'   -- or use BETWEEN for date range

▸ Key terms
event_params
Nested ARRAY<STRUCT> holding all parameters for an event. Requires UNNEST to query.
user_pseudo_id
Anonymous user ID based on cookie. Your main user identifier — not PII.
_TABLE_SUFFIX
Wildcard filter for sharded tables. Always filter this to avoid full scans and high costs.
Intraday table
Today's live streaming data. Gets replaced by the final daily table next morning.
ga_session_id
Session identifier inside event_params. Combine with user_pseudo_id for a unique session key.
⚠ BQ cost reminder
BigQuery charges per TB scanned. GA4 tables get large fast. Always use _TABLE_SUFFIX filter in every query — this limits scan to specific date tables. Querying events_* without suffix filter = scans ALL history = surprise bill. At your internship, ask the billing limit before running large queries.
